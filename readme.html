Overview

Background

The background of the project is an expansion of the synchronous, reactive, bidirectional enforcement framework.

In the below diagram, the high level architecture of the system is shown, with the Plant, Enforcer, and Controller loop. An enforcer is a formally defined construct that given a property, it will uphold that property in the system by way of transforming the input-output signals of the system. The enforcer has a notion of reject or accepting the inputs, and when it would reject them, it transforms them. For further detail into the enforcer framework, read Runtime Enforcement of Cyber-Physical Systems (P. Roop, S. Pinisetty, EMSOFT’17).


Cyber-Physical System architecture of a bidirectional enforcer.

The project expanded upon this framework by introducing priority of properties, as the enforcer upheld a single property, with many being composed together and would therefore be monolithic. The priorities allowed conflicting system behaviour, and how to state which outputs would be released to the environment.

These properties are specified as their own Synchronous Discrete Timed Automata (SDTA), and their input and outputs are chained together as shown in the diagram above.

Implementation

The system is a toolchain as seen in the below image, that takes a user defined set of properties in the DOT language, and then shows their individual diagrams and generates the C code that are the SDTAs. These SDTAs are implemented using structs and function pointers, allowing a generic enforcement algorithm.

The use case of the system is described in the Usage section below.

Toolchain

codegen.py is our program for generating C code and Graphviz visualizations from a specification source file based on graphviz. codegen.py accepts any number of source files representing a property, and uses them to generate a single file, with the name generate.c in a folder specified by a required output argument. Additionally, pdf visualizations, generated using Python’s Graphviz library, will be generated in the same output folder if the optional --visualize flag is present. Source files must have the .gv file extension. The file __sdta_template.c must be in the same directory as codegen.py and must not be modified.

For a complete description of usage, run python codegen.py --help.

An example of an input file is provided below, for the LRI property.

digraph lri {
    rankdir=LR;
    size="8,5"
    
    node [shape = doublecircle] l0;
    node [shape = circle, style=dashed] lv;
    node [shape = point] __init;
    
    __init -> l0;
    l0 -> l0 [label="VS or VP and v_lri >= 5; v_lri = 0"];
    l0 -> lv [label="v_lri > 5"];
    lv -> lv [label="true"];
    
    //__alphabet [AS, VS, AP, VP]
    //__clocks [v_lri]
    //__priority 5
    
}

To generate C code and a Graphviz visualization from the above file, place the above code in a file named lri.gv, and put this file in the same directory as codegen.py. Then run the command: python codegen.py --visualize output lri.gv.

To generate C code from DOT language, codegen.py has additional requirements for its source files. These are described below.

  1. Nodes representing accepting locations must be specified using the doublecircle shape attribute.
  2. Nodes representing the non-accepting location must be specified using the dashed style attribute.
  3. There must be a node named __init.
  4. There must be a transition from the __init node to another node.
  5. Transition conditions are specified inside the label attribute of edges, and the action and guard must be combined into a single boolean expression.
  6. The transition condition must be a valid Python boolean expression, and use only boolean operators, inequality operators, equality operators, parentheses, and clocks/signals specified below.
  7. Clock resets, if present, must be of the form clk = 0 and placed inside the label attribute of edges.
  8. If clock resets are present, a semicolon ; must separate the transition condition and clock resets within the label attribute.
  9. If multiple clock resets are present, commas , are used to delimit these.
  10. The non-accepting location must have a transition to itself, using the transition condition true.
  11. The alphabet must be declared using the form //__alphabet [ALPHABET] where ALPHABET is a comma-delimited list of signals used in transition conditions.
  12. The clocks must be declared using the form //__clocks [CLOCKS] where CLOCKS is a comma-delimited list of clocks used in transition conditions.
  13. The priority must be declared using the form //__priority PRIORITY where PRIORITY is some positive integer.
  14. The file must be a directed graph digraph.

There are no more requirements for C code generation. However, to generate a conventional visualization, users may wish to specify other attributes: for example, using the point shape attribute for __init and the circle shape attribute for all accepting locations.

This generated code can then be ran in the system by linking the files together in a makefile, as shown in Code/C/Heart_Pacemaker/Exhibition/Makefile

To run the heart, enforcer, and pacemaker in C, navigate to /Code/C/Heart_Pacemaker/Exhibition/Makefile/, and type make.

Python Enforcer

The development system is done in Python and was extended from Dr. Srinivas Pinisetty’s python implementation.
These files are located inside /Code/Python/Enforcer. The main Python file is tests/enforcer_chain_io.py, and contains the main function, as well as property generation. tests/Examples.py shows the layout for specifying new properties to run in Python. All that is required is generating the appropriate variables to run the file.

The enforcement function is inside the SyncEnforcer.py file, with automata defintions in Automata.py

Recommended Readings

With this compendium are key papers that were helpful during the course of this project in the /Papers folder.

  1. Runtime Enforcement of Cyber-Physical Systems – A Pacemaker Case Study. S. Pinisetty, P. Roop et al.
    This paper was what the work was based upon, and was the working title before submission to EMSOFT '17. This is essential to understand the system.

  2. Towards the Emulation of the Cardiac Conduction
    System for Pacemaker Testing. Eugene Yip et al.
    Background of the heart pacemaker systems, this work lead towards the 33 node heart model used in the system.

  3. Modular code generation for emulating the
    electrical conduction system of the human heart. N. Allen et al.
    Paper about the generation of the heart into C code as used in the system.
